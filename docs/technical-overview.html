Procurement Knowledge Base — Pipeline & MCP Server

A local data pipeline and MCP server that let an AI agent retrieve, compare and reason across a heterogeneous procurement knowledge base — invoices, purchase orders, shipping orders, inventory reports and a supply contract — with answers grounded in source references.

Miguel Ángel Santana Hernández  ·  Senior AI Data Software Developer  ·  June 2026
The design in one line: the data has two shapes, so retrieval has two shapes. The transactional records (invoices, POs, shipping, inventory) are relational and join on an order_id, so they are answered with deterministic SQL. The single narrative document (the contract) is the only thing embedded for semantic search. The MCP client is the agent; the server is a grounded, auditable tool layer.

Part IThe system at a glance

1What this is, and the problem it solves

The brief provides a small procurement knowledge base — 45 files across five categories (invoices, purchase orders, shipping orders, inventory reports, contracts) — as a mix of clean PDFs, a long contract, and scanned image invoices. The task is a local pipeline that prepares those files plus an MCP server that lets an AI agent (Claude Desktop, ChatGPT, any MCP client) ask questions over them and answer with citations.

The interesting part is not parsing files; it is the architecture. Looking at the example questions, seven of the eight are relational — "which invoices are missing a purchase order", "which documents support order 10687", "are there mismatches between invoice, PO and shipping". Only one is genuinely semantic — "summarize the contract terms relevant to supply of goods". So the obvious move, "embed every document into a vector store", would be the wrong architecture: vector similarity cannot prove a row is absent, and it cannot guarantee an exact lookup. The design follows the data instead.

2Architecture overview

Three layers, one database, no external services. A pipeline prepares the files into a single SQLite database (relational tables + a keyword index + the contract vectors). A FastMCP server exposes six typed tools over those tables. The MCP client does the reasoning and calls the tools. The whole knowledge base is one ~2 MB file and one process — for 45 documents, that is the right amount of machinery.

Sources → one extractor per shape → a single SQLite database → six MCP tools → the client routes and answers with citations. The image invoices (amber) stay on their own path and never join the order graph.

Key technologies: Python (managed with uv) · SQLite + sqlite-vec · FTS5 · fastembed (bge-small-en-v1.5, 384-d) · pdfplumber · PyMuPDF · Anthropic vision · the official MCP SDK (FastMCP). Eight direct dependencies, no GPU, no torch, no external services — detailed in §13.

3How an AI agent uses it (interaction flow)

This is the part that makes it an agentic system rather than a search box. The orchestration is not hard-coded; the MCP client is the agent, and it decides what to call.

  1. The user asks a natural-language question in the client (e.g. Claude Desktop).
  2. The client reads the tool descriptions and typed schemas the server advertises, and picks the tool whose contract matches the question. For "missing a PO" it picks reconcile; for the contract it picks search_contract.
  3. It calls the tool over stdio (JSON-RPC). The server runs a read-only query and returns structuredContent — a validated object, not prose — including SourceRefs.
  4. The client may chain tools (e.g. get_order_dossier → get_document), then composes the final answer and cites the sources.

The server therefore stays a deterministic capability layer. There is no second agent loop inside it — that would be redundant behind the one already calling it. The design lever is the tool surface: name and type the tools well, and the right one becomes the obvious choice for the model.

Part IIThe data

4The data: three shapes

Before any code, the files sort into three shapes, and the whole architecture follows from that reading:

ShapeFilesNatureConsequence
Transactional recordsinvoices, POs, shipping orders, inventory reports (Northwind PDFs)structured / semi-structured, consistent templates, all keyed by order_idparse deterministically → relational tables → SQL
One narrative documentthe 71-page supply contract (digital PDF)unstructured legal prosechunk + embed → the only thing in the vector index
Scanned image invoices5 JPGs (two visibly different layouts)multi-modal, heterogeneous, a different vendor schema entirelyvision-LLM extraction → a separate, tolerant schema

Two findings inside the data shaped specific decisions, covered later: the image invoices are not even one schema among themselves (§7), and the contract is a blank template whose highlight colors carry meaning (§6, §8).

5Data flow (the pipeline)

One re-runnable pipeline (python -m procurement_kb.pipeline.run) walks data/ and builds procurement.db in five steps:

  1. Register — every file is recorded in documents (SHA-256, category, file type, page count) so every later record has lineage back to a source file.
  2. Route by shape — clean Northwind PDFs to deterministic parsers; the 5 image invoices to the vision model; the contract to PyMuPDF.
  3. Extract — structured fields and line items for the records; text + highlight-color metadata for the contract; a tolerant structured object for each image invoice.
  4. Index — records into relational tables and a keyword index (doc_fts, BM25); contract chunks into contract_fts (BM25) + contract_vec (384-d vectors).
  5. Serve — the MCP server then runs read-only queries behind six tools. The MCP layer consumes processed data; it never extracts at request time.

Reproducible & re-runnable. The database is dropped and rebuilt each run (child tables before parents, so foreign-key checks pass on a re-run over a populated DB), so row counts are stable and there is no partial-state drift (idempotent). Only the non-deterministic vision step is cached — by content hash of the image bytes — so the deterministic parsers always run fresh and can never go stale, while the whole pipeline still runs offline with no API key (the 5 extractions ship pre-computed in cache/). Because the cache key is the image's content hash, a re-run re-extracts only images that are new or changed (one Claude-vision call each); unchanged images come from cache/ with no API call. Dependencies are pinned with uv.lock. The vision extraction is forced through a structured tool call, so its output always conforms to the schema and can be safely cached.

Seeing a data change in the client. The MCP server reads only procurement.db — never data/ or cache/ directly. So after you add, change, or remove a file in data/, re-run the pipeline to rebuild procurement.db; the client then reflects it on its next tool call automatically — no Claude Desktop restart needed. (Editing cache/ alone changes nothing the client sees; the cache is only an input to the pipeline.) Restart Claude Desktop only when you change the server code or the config, which it reads at startup.

6Data modeling choices

order_id is the foreign key of the transactional graph: invoices, purchase_orders and shipping_orders all join on it; inventory_reports join on period and category, not on an order. Each of the three order tables has its own 1—N line-item child table. A documents table is the lineage registry, so every record maps back to a source file and page.

The five image invoices are modeled as a separate schema (image_invoices): they carry no order_id and are never joined to the order graph, because they genuinely are not related — different vendors, different layouts, a VAT/IBAN invoice format that has nothing to do with the Northwind orders. Forcing them into the order graph would manufacture false matches; keeping them as their own island is the honest model. The contract is stored as contract_chunks with a highlight column derived from its template colors (§8).

7Structured extraction vs document retrieval

A central decision was per shape: extract structured fields, or retrieve text? The answer is different for each source, and the reasoning is the point.

SourceApproachWhy
Northwind PDFsstructured extraction, deterministic parserconsistent templates → a small parser is exact, auditable, free, and never hallucinates. An LLM here would add cost and variance for no gain.
Image invoicesstructured extraction, via a vision LLMthis is the one place that genuinely needs AI: the layouts vary, so a template parser would break. The LLM absorbs layout variance into a typed schema.
Contractdocument retrieval (chunk + embed)narrative prose with no fixed fields; the question is semantic ("supply terms"), so chunked hybrid search is the right tool.

The image invoices are heterogeneous even among themselves — some are formal invoices with a header, IBAN and VAT; others are bare industrial invoices with no header at all, and even differ from each other (one carries a sales tax line, another does not). A rigid template parser would break on that, so they share one tolerant schema (most fields Optional) with a layout_family tag, and the vision model absorbs the variance. The extraction is forced through a structured tool call (so the output always conforms to the schema) and cached by content hash, so it is stable and offline-reproducible.

Part IIIRetrieval & the MCP surface

8Retrieval strategy

Most of the example questions are relational, not semantic. Vector similarity returns ranked, approximate neighbours — it cannot prove the absence of a matching row, or guarantee an exact lookup. So those questions are answered with SQL, and only the contract is embedded.

Question typeOperationTool
invoices missing a POanti-join / set differenceSQL (reconcile)
which documents support an orderexact lookupSQL (get_order_dossier)
mismatches across invoice / PO / shippingmulti-table join + compareSQL (reconcile)
evidence about a vendor / itemkeyword + filterFTS5 (search)
inventory available / periodmetadata querySQL (list_inventory_reports)
summarize contract supply termssemanticvector + BM25 (search_contract)

For example, "which invoices are missing a purchase order" is a set difference — exact, explainable and auditable, which a similarity score is not:

SELECT order_id FROM invoices
WHERE order_id NOT IN (SELECT order_id FROM purchase_orders);
-- → 10436, 10687, 10839

Contract search is hybrid

Dense vectors (sqlite-vec) and BM25 keyword search (FTS5) run in parallel and are fused with Reciprocal Rank Fusion (k=60). They are complementary: the dense vector captures meaning and paraphrase, while BM25 catches the exact terms a dense vector blurs — clause numbers, defined terms, party names. RRF combines their rankings without needing the two scores to be on the same scale.

The contract's colors are data quality

The contract is a blank template; its highlight colors are a legend it defines itself (yellow = placeholder, green = authoring instruction, grey = optional clause). These are vector rectangles in the PDF, read directly — no OCR, zero tokens. They become a highlight column used as a data-quality signal: the green authoring-instruction block (the document's own "to be deleted before issuing" note) is excluded from the index, and yellow placeholders are flagged, so the agent never returns template boilerplate (for example [NAME OF SUPPLIER]) as if it were a real, executed term. This is the kind of validation an audit platform depends on.

9The MCP tools, in detail

A small, typed tool set — six tools. Each returns a Pydantic model, so the SDK emits structuredContent with an outputSchema the client can rely on. Each returns ranked hits with snippets rather than raw dumps (so the agent does not drown in unnecessary data), and each carries one or more SourceRef. The client decides which tool to call.

search(query: str, doc_type?: str, limit: int = 5) → SearchResults

Keyword evidence across every record type

How it works: tokenizes the query and runs an FTS5 MATCH over doc_fts (a keyword index spanning all structured records), optionally filtered by doc_type (e.g. only invoices), ordered by BM25 rank, capped at limit.
Returns: { query, count, results[] } where each hit is { doc_id, category, order_id?, snippet, source } — the snippet is a short bracketed excerpt around the match, not the full document.
Answers: "find evidence related to a specific vendor, order, or item."
get_document(doc_id: str) → DocumentOut | None

The full structured record for one document

How it works: looks up the document's category in documents, then routes to the matching table and returns its fields plus the 1—N line items. One tool covers all six categories. For the contract it returns a small note and chunk counts instead of dumping 71 pages, pointing the agent to search_contract.
Returns: { doc_id, category, fields{}, line_items[], source }, or None if the id does not exist.
Answers: "show me the full invoice / PO / shipping order / inventory report / image invoice."
get_order_dossier(order_id: int) → OrderDossier

Which documents support a given order

How it works: for the order, checks the presence of an invoice, a purchase order and a shipping order; for each present document returns its doc_id, a small summary (key columns) and a SourceRef; a note states what, if anything, is missing.
Returns: { order_id, invoice, purchase_order, shipping_order, note }, each sub-object being { present, doc_id?, source?, summary{} }.
Answers: "which documents support order 10687?" and "what purchase order supports this invoice?"
reconcile(order_id?: int) → list[Reconciliation]

Three-way cross-document match (the core reasoning tool)

How it works: for one order — or for every order in the knowledge base when called with no argument — it checks the presence of invoice / PO / shipping, then compares line items across whatever documents exist, keyed by product name: it flags any product whose quantity differs between documents, and any whose unit price differs (compared to two decimals). It returns a status of complete, missing_document, or quantity_or_price_mismatch with human-readable findings.
Returns: per order, { order_id, has_invoice, has_purchase_order, has_shipping_order, status, findings[], line_diffs[], sources[] }.
Answers: "are there mismatches between invoices, POs and shipping?", "does this shipment match the related invoice?", and "which invoices are missing a matching purchase order?"
search_contract(query: str, k: int = 5) → list[ContractHit]

Hybrid semantic + keyword search over the contract

How it works: embeds the query with fastembed, runs a KNN search over the contract vectors (with a numpy cosine fallback if sqlite-vec is unavailable) and a BM25 search over contract_fts in parallel, then fuses the two rankings with Reciprocal Rank Fusion (k=60). Returns the top-k chunks with their section heading, page, the highlight tag, and a fused score.
Returns: a list of { chunk_id, section, snippet, highlight, score, source(+page) }; highlight is one of none / placeholder / option.
Answers: "summarize the contract terms relevant to supply of goods."
list_inventory_reports() → list[InventoryReportSummary]

What inventory data exists, and the period it covers

How it works: lists the inventory reports with the count of their line items, ordered by period — a pure metadata query, no parameters.
Returns: a list of { doc_id, period, category, category_id?, product_count, source }.
Answers: "what inventory reports are available, and what period do they cover?"

Example — reconcile(order_id=10687) returns typed, cited output:

{ "order_id": 10687,
  "has_invoice": true, "has_purchase_order": false, "has_shipping_order": true,
  "status": "missing_document",
  "findings": ["No purchase order found for order 10687."],
  "sources": [
    { "doc_id": "invoice_10687", "category": "invoice",
      "source_path": "data/invoices/invoice_10687.pdf" },
    { "doc_id": "order_10687", "category": "shipping_order",
      "source_path": "data/shipping_orders/order_10687.pdf" } ] }

No agent framework runs inside the server. In MCP the client is the agent; embedding a second agent loop in the server would be redundant.

10Citation / source-reference strategy

Grounding is structural, not bolted on. Every tool result carries a SourceRef { doc_id, category, source_path, page }; contract hits also include the section heading and page. Behind them, the documents table maps every record back to its original file (with its SHA-256). So each claim in an answer traces to a specific document and, for the contract, a specific page.

12Tech stack: dependencies and why each was chosen

Eight direct dependencies, all local. The bar for each was "the lightest tool that is actually reliable for this shape of data" — no GPU, no torch, no running services, no cloud.

LibraryRole hereConsidered insteadWhy this one
mcp[cli]
FastMCP		 the MCP protocol — server, stdio transport, typed tools a hand-rolled JSON-RPC server; a heavier agent framework official SDK; typed decorators auto-emit structuredContent + outputSchema; ships the Inspector/CLI for testing
pdfplumber deterministic parsing of the Northwind PDFs (text + tables) pypdf (text only, weak on tables); camelot / tabula (heavier; tabula needs Java); unstructured (heavy) pure-Python, reliable positional text and table extraction, light footprint
pymupdf
(fitz)		 contract text + highlight colors as vector rectangles, no OCR pdfminer / pdfplumber (awkward for fill colors); Tesseract / Textract OCR (unnecessary — the PDF is digital) fast, and exposes the vector draw layer so the colors are read with zero OCR and zero tokens (note: AGPL — fine for a local tool)
fastembed local embeddings — bge-small-en-v1.5, 384-d, ONNX sentence-transformers (pulls in torch, ~GBs); OpenAI / Cohere embeddings (API key + network) ONNX runtime — no torch, no GPU, no key; compact 384-d vectors; runs fully offline
sqlite-vec vector KNN inside the same SQLite file FAISS (separate index); Chroma / Qdrant / Weaviate (a running service); pgvector (needs Postgres) lives inside the one database file, zero services; ample for a few hundred chunks; numpy cosine fallback included
anthropic Claude vision structured extraction for the 5 image invoices Tesseract OCR (brittle on heterogeneous layouts); AWS Textract (cloud); docling + TableFormer (local but pulls torch) absorbs layout variance into a typed schema; forced structured tool call; the result is cached so the pipeline stays offline
pydantic typed extraction records + typed tool I/O dataclasses (no validation); attrs; marshmallow the MCP SDK uses it to emit structuredContent + outputSchema; validates extraction at the boundary
numpy cosine-similarity fallback when sqlite-vec can't load pure-Python math (slow) already transitive; keeps vector search working everywhere with no extra weight

Standard library does the rest: sqlite3 (storage + FTS5 keyword index), hashlib (SHA-256 for lineage and the cache key), logging — routed entirely to stderr, never stdout, because stdout is the MCP JSON-RPC channel and a stray print would corrupt it — plus re, pathlib and json. Managed with uv; versions pinned in uv.lock.

14Install, run & connect to Claude Desktop

Everything needed to go from a clone to live tool calls in Claude Desktop, in one place — you should not need any other file to install this. Prerequisites: uv and Python ≥ 3.11. No API key is needed — the 5 image extractions ship pre-computed in cache/, so everything runs offline.

Step 1 — install dependencies and build the database

# create the venv + install locked dependencies
uv sync

# build the knowledge base → procurement.db (re-runnable, idempotent)
uv run python -m procurement_kb.pipeline.run

Step 1b — (optional) an Anthropic API key, for new / changed images

The default build needs no key (the 5 image extractions ship in cache/). The key is consulted only on a cache miss — a new image, a changed image, or an empty cache/ — because the vision extraction is cached by the image's content hash. Provide it any of three ways (first match wins): a .env file at the repo root (auto-loaded, git-ignored), a shell export, or inline. PKB_VISION_MODEL (default claude-opus-4-8) overrides the vision model.

# (a) .env at the repo root  ← recommended; auto-loaded, git-ignored
cp .env.example .env          # then set ANTHROPIC_API_KEY=sk-ant-...
# (b) export ANTHROPIC_API_KEY=sk-ant-...
# (c) ANTHROPIC_API_KEY=sk-ant-... uv run python -m procurement_kb.pipeline.run

Because the pipeline rebuilds from data/ every run and caches by content hash, you can demonstrate the full extraction flow without writing any code: clear cache/ (then run) to re-extract all 5 live; add or edit a JPG in data/invoices/ and re-run, and only the new / changed image calls the API while the rest load from cache; remove a JPG and it simply drops out of the next build.

# full live extraction of all 5 (no cache):
mv cache cache.bak && uv run python -m procurement_kb.pipeline.run   # 5 vision calls
rm -rf cache && git checkout -- cache                              # restore committed cache

# add / change an image, then re-run — only that file calls the API:
cp my_invoice.jpg data/invoices/ && uv run python -m procurement_kb.pipeline.run

Step 2 — (optional) verify before wiring up a client

# 8 example questions → grounded answers, offline, no GUI
uv run python scripts/smoke_test.py

# a real MCP protocol round-trip over stdio
uv run python scripts/mcp_client_test.py

# or inspect the tools visually in a browser, no install
npx @modelcontextprotocol/inspector uv run python -m procurement_kb.server.mcp_server

Step 3 — register the server in Claude Desktop

You don't run the server yourself — Claude Desktop launches it over stdio once it's registered.

  1. Open the config file (create it if it doesn't exist) — Claude Desktop → Settings → Developer → Edit Config, or open it directly:
    • macOS: ~/Library/Application Support/Claude/claude_desktop_config.json
    • Windows: %APPDATA%\Claude\claude_desktop_config.json
  2. Add the procurement-kb block inside mcpServers, replacing ABSOLUTE_PATH with the full path to this repo. If you already have other MCP servers, add it alongside them inside the same mcpServers object: 
    {
  "mcpServers": {
    "procurement-kb": {
      "command": "uv",
      "args": [
        "--directory",
        "ABSOLUTE_PATH/caseware-procurement-mcp",
        "run",
        "python",
        "-m",
        "procurement_kb.server.mcp_server"
      ]
    }
  }
}
  3. If Claude Desktop can't find uv — the server shows as failed, or the log says spawn uv ENOENT — Claude Desktop does not inherit your shell PATH, so a bare "command": "uv" fails when uv isn't on a standard system path. Run which uv (macOS/Linux) or where uv (Windows) and use that absolute path as "command", e.g. "command": "/Users/you/.local/bin/uv".
  4. Fully quit and reopen Claude Desktop (⌘Q on macOS — the config is read only at startup).

Step 4 — confirm it's connected

The server appears under Settings → Developer and the six tools show in the tools (🔌) menu. Ask an example question — "Which invoices are missing a purchase order?" — and Claude calls the tools and answers with sources.

Troubleshooting: if the server doesn't appear, the log says why — ~/Library/Logs/Claude/mcp-server-procurement-kb.log (macOS). The usual cause is the uv path (step 3) or a wrong ABSOLUTE_PATH. The same procurement-kb block also works in any other MCP client — the server speaks plain MCP over stdio.

16AI-assisted development & validation

Built with AI assistance (Claude Code) for scaffolding, the deterministic parsers, and the MCP wiring — the brief requires using AI-assisted tooling. The architecture and trade-offs are my own. Validation was layered: extraction was cross-checked against the raw PDF text and the source images; smoke_test.py and mcp_client_test.py verify the answers, the sources, and the MCP round-trip; the image extraction is constrained to the schema via a forced structured tool call and cached; and every SQL query behind a tool is auditable by hand. Where a result depends on the model (the 5 image invoices), it is the only AI-extracted data and is isolated behind its own cached, validated schema.

17Scaling path (production, AWS)

The same shape maps onto managed services without changing the design: S3 for ingestion, Textract (AnalyzeExpense) or Bedrock Data Automation for extraction, Step Functions / Lambda for orchestration, Aurora + pgvector / OpenSearch Serverless / S3 Vectors for storage, Bedrock Knowledge Bases for retrieval with citations, Claude on Bedrock for reasoning, and the MCP server hosted on Bedrock AgentCore (Runtime, or tools via Gateway). Governance for an audit customer comes from CloudTrail (audit trail), DataZone (lineage), and Bedrock Guardrails — with Bedrock not training on customer data. The local prototype already mirrors the structure: lineage table, deterministic extraction, citations, and a typed tool boundary.
